# 審査員向け画面録画シナリオ / Screen Recording Script for App Review

> Meta の App Review では「機能デモの画面録画」提出が求められます。
> このスクリプトに沿って録画してください。録画時間の目安：**5〜8分**

---

## 録画前の準備

| 準備物 | 確認 |
|--------|------|
| テスト用 Threads アカウント（テスターとして登録済み） | |
| テスト用投稿（下記サンプル投稿を事前に投稿しておく） | |
| Google Sheets（THREADS_MANUAL_REPLY_INBOX）をブラウザで開いておく | |
| Cloud Run エンドポイント URL を手元に用意 | |
| LINE 通知確認用スマートフォン | |

### テスト用サンプル投稿（テストアカウントで事前投稿）

投稿1（TACHINOMIYA 向け）:
```
沖縄旅行中です！国際通り周辺で今夜ひとりで気軽に食べられるお店を探しています。
沖縄料理が食べたいです。おすすめ教えてください！
```

投稿2（琉球火鍋向け）:
```
来週那覇で友達と女子会をします。個室で落ち着いてしゃぶしゃぶか火鍋が
食べられるお店を探しています。おすすめありますか？
```

## 録画シナリオ（ナレーション付き）

### シーン 1：システム概要の説明（30秒）

> 「これは Trees Hospitality Connect のデモです。沖縄の飲食店2店舗、
> TACHINOMIYA と琉球火鍋に向けて、Threads 上で飲食店を探しているユーザーを
> 発見し、スタッフが手動返信するための支援ツールです。」

**画面**: システム構成図または `docs/threads_app_review_submission.md` を表示

---

### シーン 2：キーワード検索の実行（1分）

> 「threads_keyword_search API を使って、沖縄グルメ関連のキーワードで検索します。」

**操作手順**:
1. ブラウザで Threads Graph API Explorer を開く
2. 以下のリクエストを実行する（テスト用アクセストークンを使用）：

```
GET https://graph.threads.net/v1.0/keyword_search
  ?q=沖縄+おすすめ+教えて
  &search_type=RECENT
  &access_token={テスト用トークン}
```

3. レスポンス（JSON）を画面に表示する
4. 投稿リストが返ってきていることを確認

> 「検索結果には投稿 ID・テキスト・ユーザー名・投稿 URL が含まれています。
> プライベート投稿は含まれません。」

---

### シーン 3：AI 関連性スコアリング（1分）

> 「取得した投稿はAIが100点満点でスコアリングします。
> 5つの判定基準があります。」

**画面**: スコアリング基準の説明スライドまたは以下のテキストを表示

```
スコアリング基準（100点満点）
① 沖縄への旅行中・旅行予定       : 25点
② 飲食店を探しているという意図    : 25点
③ 那覇・国際通りエリアへの言及   : 20点
④ 当店舗の特徴との一致           : 20点
⑤ 日付・人数・シーン等の具体性   : 10点

通知閾値: 70点以上のみスタッフへ通知
```

**操作手順**:
1. Cloud Run テストエンドポイントを呼び出す：

```bash
curl -X POST https://tachinomiya-ai-75610219333.asia-northeast1.run.app/threads-manual-test
```

2. レスポンス JSON を表示
3. 各テスト投稿のスコアと推奨店舗を確認

> 「テスト投稿1は75点でTACHINOMIYA向けと判定されました。
> テスト投稿2は80点で琉球火鍋向けと判定されました。」

---

### シーン 4：スプレッドシートへの記録（1分）

> 「判定結果はすべて Google Sheets に記録されます。」

**操作手順**:
1. Google Sheets（THREADS_MANUAL_REPLY_INBOX）を開く
2. 新しく追記された行を表示
3. 各列（投稿URL・スコア・推奨店舗・返信案・通知状況）を説明

> 「投稿 URL、スコア、推奨店舗、AI が生成した返信案、通知状況が
> それぞれの列に記録されています。スタッフの対応状況も管理できます。」

---

### シーン 5：スタッフへの LINE 通知（30秒）

> 「スコア70点以上の投稿は、スタッフの LINE に通知が届きます。」

**操作手順**:
1. スマートフォンの LINE を開く
2. 届いた通知メッセージを表示
3. 通知内容（推奨店舗・スコア・投稿URL・返信案・スタッフ作業手順）を読み上げ

> 「推奨店舗、関連性スコア、投稿URL、AI生成の返信案が含まれています。
> スタッフはこの通知を見て、投稿を自分で確認します。」

---

### シーン 6：スタッフによる手動返信（1分）

> 「実際の返信は、スタッフが Threads アプリを開いて手動で行います。
> 自動返信は一切ありません。」

**操作手順**:
1. スマートフォンまたはブラウザで Threads を開く
2. 通知に含まれていた投稿URLを開く
3. 返信案をコピーして返信フォームに貼り付ける（テスト投稿への返信のため実施可）
4. 投稿内容を確認・必要に応じて編集する
5. 「返信」ボタンをタップ

> 「スタッフが投稿の内容を確認し、返信案を必要に応じて修正してから
> 手動で返信します。システムが自動的に投稿することはありません。」

---

### シーン 7：重複防止の確認（30秒）

> 「同一投稿への重複返信を防ぐ機能をご覧ください。」

**操作手順**:
1. 処理エンドポイントを再度呼び出す
2. 同じ投稿が「重複URL」として除外されることを確認：

```json
{"status": "除外", "reason": "同一URLが既に通知済みです"}
```

> 「同一の投稿は2回通知されません。また、同一ユーザーへの通知は
> 7日間のクールダウンが設定されています。」

---

### シーン 8：DRY_RUN モードの説明（30秒）

> 「本番環境では DRY_RUN=true が設定されており、
> システムが自動的に Threads へ投稿することは絶対にありません。」

**操作手順**:
1. DRY_RUN ステータス確認エンドポイントを呼び出す：

```bash
curl https://tachinomiya-ai-75610219333.asia-northeast1.run.app/threads-dry-run-status
```

2. レスポンスを表示：

```json
{"dry_run": true, "env_value": "true", "note": "App Review承認前は DRY_RUN=true のまま運用してください"}
```

> 「DRY_RUN フラグが true になっており、投稿 API は呼び出されません。
> App Review 承認後も、スタッフ承認なしの自動返信は行わない設計です。」

---

### シーン 9：シートの対応状況更新（30秒）

> 「スタッフが手動返信を完了したら、シートに記録します。」

**操作手順**:
1. Google Sheets を開く
2. 対応した行の「スタッフ対応状況」列を「返信完了」に変更
3. 「対応者」列にスタッフ名を入力

> 「これにより、どの投稿に誰が対応したかが記録として残ります。
> 同一投稿への二重対応も防止できます。」

---

## 録画のポイント

- 画面上に個人情報（本名・実際のメールアドレス等）が映らないよう注意
- テスト用アクセストークンは録画後すぐに失効させること
- 録画ファイル形式：MP4 推奨、5〜8分、1080p 以上
- 字幕がない場合はナレーションを明瞭に録音すること
- 英語ナレーションが望ましい（日本語でも可、その場合は英語字幕を追加）
