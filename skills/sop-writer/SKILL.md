# sop-writer（SOP変換・委任指示Skill）

## 目的
「ゆうさんの頭の中の決定」を、**担当者が質問ゼロで実行完了できる文書**に変換する。
AIが指示内容を6〜7割作り込んでシンプルに渡す標準（YU HOLDINGS運用原則）の実行装置。
ゆうさんの確認ポイントを最小化し、「全部ゆうさんに聞く」構造を壊す。

## 対象（受け手）
| 受け手 | 例 | 文体基準 |
|---|---|---|
| TACHINOMIYA現場スタッフ（ゆうと他） | 口コミ依頼徹底・撮影・仕込み | 短文・チェックボックス型 |
| Catering 天久/外注 | 事例写真整理・見積実務・配送手配 | 手順＋判断基準明記 |
| Tree Beauty業務委託（美優） | IG運用・施術写真補充 | 手順＋NG例重視（薬機法） |
| SNS/撮影/Drive整理担当 | 投稿スクショ報告・写真分類 | 画面ベース・具体名 |
| 営業外注 | DM送信前リスト作成 | 数値基準＋報告様式厳格 |
| AIネットビジネス運用担当 | Threads返信・購入者対応 | テンプレ参照型 |

## 使う場面
- ゆうさんが決定を下した直後（決定→SOP化を即時セットで）
- 同じ質問がスタッフから2回来たとき（=SOP不在のシグナル）
- 新規外注の受け入れ（依頼文＋SOPを同時生成）
- 既存業務の委任範囲拡大（ゆうさん確認箇所の削減設計）

## 入力スキーマ
```yaml
sop_request:
  decision: string          # ゆうさんの決定・指示（口語のままでOK）
  assignee: string          # 担当者（不明なら候補を出させる）
  business: string
  deadline: string|null     # 期限（なければSkillがデフォルト提案）
  frequency: string         # 単発 / 毎日 / 毎週◯曜 / 都度
  output_type: full_sop | short_instruction | outsource_request
  context: string|null      # 背景・過去の失敗・こだわり
```

## 判定ロジック（SOP品質の必須7要素）
生成するSOPは以下7要素が**全て埋まっていること**。1つでも埋められない場合は、勝手に省略せず「ゆうさんへの確認質問」として明示する。
1. **担当者**（名前。 「スタッフ」「誰か」は不可）
2. **期限**（日付 or 頻度。「なるべく早く」は不可）
3. **手順**（1手順=1動作。画面操作は具体名「Driveの【フォルダ名】を開く」）
4. **完了条件**（何がどうなったら終わりか、目で見て判定できる形）
5. **NG例**（過去の失敗 or 予見される間違いを最低2つ）
6. **報告方法**（誰に・何を・どの形式で。写真/スクショ報告の要否を明記）
7. **ゆうさん確認ポイント**（最大2箇所まで。3箇所以上になる場合は委任設計を見直す提案を添える）

**専門用語の変換規則**
- GCS化→「クラウド保存（担当◯◯に渡すだけでOK）」等、受け手が実行に必要なレベルまで平易化
- ただしツール固有名（Drive・Airレジ・Canva）はそのまま使用（現場の共通語）

## 実行手順
1. decisionを分解：やること／判断が要ること／ゆうさんしか決められないこと に3分類
2. 「ゆうさんしか決められないこと」が3つ以上→委任設計の見直し提案を先に出す
3. 必須7要素を埋める。埋まらない要素は確認質問化（最大3問）
4. output_typeに応じて出力：full_sop（手順書）／short_instruction（LINE1通で送れる短文）／outsource_request（外注依頼文＝金額・納期・検収条件つき）
5. 報告テンプレを必ず付ける（受け手がコピペで報告できる形）
6. 末尾に「このSOPで運用開始しますか？ Yes/No」

## 出力スキーマ
```yaml
result:
  sop:
    title: string           # 作業名（動詞で終わる。例:「サーターアンダギー画像を撮影してDriveに入れる」）
    purpose: string         # 1行。売上/集客/稼働ゼロ/売却価値のどれに効くか明記
    assignee: string
    deadline: string
    frequency: string
    required_items: [string]
    steps: [string]         # 1手順=1動作
    checklist: [string]
    photo_report_required: bool
    completion_criteria: string
    ng_examples: [string]   # 最低2つ
    report_template: string # コピペ可能な報告文
    owner_check_points: [string]  # 最大2つ
  short_instruction: string|null  # LINE用短文（output_type該当時 or 併記）
  outsource_request: string|null  # 外注依頼文（金額欄は【ゆうさん承認後】）
  open_questions: [string]|null   # ゆうさんへの確認（最大3問）
  yes_no_question: string
```

## 禁止事項
- 曖昧な指示のまま出力すること（「きれいに」「ちゃんと」「なるべく」禁止。数値・具体名に変換）
- 担当者不明・期限なし・完了条件なしのSOP
- ゆうさん確認ポイントが3箇所以上のSOP（委任になっていない）
- 受け手が知らない専門用語の多用
- 外注依頼文への確定金額の記載（金額はゆうさん承認プレースホルダ）
- 報告テンプレの省略

## 人間確認が必要なポイント
- 外注依頼の金額・納期の最終確定
- 新規担当者への初回委任（力量とSOP難易度の整合）
- お客様に直接影響する手順（口コミ依頼・DM送信等）の文言最終確認
- SOP同士の矛盾（既存SOPと手順が食い違う場合）

## 成功条件
- SOPを渡した後、担当者からの質問がゼロ〜1回で完了する
- 同じ作業の2回目以降がゆうさん関与ゼロで回る
- SOPが蓄積され、新人・外注の受け入れコストが下がる（売却時の運用資産になる）

## 失敗条件
- 「適宜対応してください」等の判断丸投げ表現が残る
- 報告が来ない/形式がバラバラ（報告テンプレの不備）
- ゆうさんへの確認が結局5回以上発生する

## 入力例
```yaml
sop_request:
  decision: "サーターアンダギーの写真が足りないから、ゆうとに撮ってDriveに入れといてほしい。自動投稿で使うやつ"
  assignee: "ゆうと"
  business: "TACHINOMIYA"
  deadline: null
  frequency: "単発（在庫不足時は都度）"
  output_type: full_sop
  context: "過去に暗い写真・横向き写真が使えなかった"
```

```yaml

    title: "サーターアンダギー画像を5枚撮影してDriveに入れる"

```
